════════════════════════════════════════════════════════════════

Columns: sensor_id, trip_id, timestamp, elapsed_seconds,
         accel_x, accel_y, accel_z, speed_kmh, gps_lat, gps_lon

CRITICAL: Only 30 of 220 trips (14%) have sensor data.
          Accel and audio cover the exact same 30 trips.

Readings per trip: min=4, max=21 (sparse — not continuous 50Hz telemetry)
Sampling interval: ~30 seconds based on elapsed_seconds gaps

Speed range: 0–62 km/h, avg=29 km/h
accel_x (lateral):    0.10–6.80, avg=1.59
accel_y (longitudinal): 0.10–5.40, avg=1.06
accel_z (vertical):   4.80–9.80, avg=9.10  (gravity ~9.8 m/s²)

Δ-accel magnitude (sqrt(x²+y²+(z-9.8)²)) — gravity-removed:
  0–2 m/s²  : 115 readings  (normal driving)
  2–4 m/s²  : 120 readings  (moderate events)
  4–6 m/s²  : 3  readings   ← potential harsh events
  6–8 m/s²  : 2  readings   ← harsh events
  8+ m/s²   : 3  readings   ← very harsh / crash-level

Threshold recommendations:
  moderate_brake  : 2–4 m/s²
  harsh_braking   : > 4 m/s²  (8 readings = 3.3% of all readings)
  extreme event   : > 6 m/s²  (5 readings = 2.1%)

Note: accel_z min is 4.8 (phone tilted / not flat on dashboard).
      z-axis alone unreliable; use full magnitude.

KEY DESIGN DECISIONS & OPEN QUESTIONS
════════════════════════════════════════════════════════════════

1. SENSOR SPARSITY (most important constraint)
   Only 30/220 trips (14%) have sensor data. The system must handle
   trips without sensor data gracefully — no flags, show "no data" state.
   Do NOT generate flags for trips without sensor readings.

2. ACCEL THRESHOLD
   Use Δ-accel magnitude (gravity-removed):
     moderate_brake  : 2.0–4.0 m/s²
     harsh_braking   : > 4.0 m/s²
   Consider: rolling window of 2+ consecutive readings above threshold
   to reduce single-sample noise.

3. AUDIO THRESHOLD
   classification ∈ {very_loud, argument} → always flag
   sustained_duration > 60s              → escalate severity
   dB > 82 as a numeric fallback if classification unavailable

4. COMBINED SCORE FORMULA (from reference output)
   combined_score ≈ weighted avg of motion_score and audio_score
   Suggested: combined = 0.6 * motion_score + 0.4 * audio_score
   (motion slightly weighted higher; tune based on reference outputs)
   Severity: low < 0.50, medium 0.50–0.70, high > 0.70

5. EARNINGS VELOCITY UX
   Early-shift velocity is inflated. Show forecast only after ≥3 trips
   or ≥1 hour elapsed. Use smoothed rolling velocity rather than
   instantaneous.

6. GOAL STATUS vs FORECAST MISMATCH
   29 drivers have "at_risk" status but "ahead" forecast — these are
   good candidates to highlight as "you turned it around" moments.
   Conversely, drivers who were "on_track" but ended "at_risk" show
   where the forecast failed.

7. TRIP QUALITY RATING LOGIC
   Quality doesn't purely track stress score. Likely a composite of:
     - stress_score (lower = better)
     - earnings_velocity (higher = better)
     - flagged_moments_count (lower = better)
   Suggested rule:
     excellent: stress < 0.3  AND no high-severity flags
     good:      stress < 0.5  OR  no high-severity flags
     fair:      stress 0.5–0.7
     poor:      stress > 0.7  OR  high-severity flag present

8. DATA JOIN KEYS
   trips ←→ accelerometer/audio: via trip_id
   trips ←→ drivers: via driver_id
   drivers ←→ driver_goals: via driver_id
   drivers ←→ earnings_velocity_log: via driver_id
   trips ←→ flagged_moments: via trip_id
   trips ←→ trip_summaries: via trip_id
   All joins are clean (no orphan IDs found).
